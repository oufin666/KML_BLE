# 硬件通信协议文档

## 1. 协议概述

本协议定义了小程序与硬件端之间的数据传输格式，硬件端期望接收**二进制数据**而非JSON文本。数据传输采用串口通信，以特定的结束序列作为传输完成标志。

## 2. 数据格式

### 2.1 整体结构

| 字段 | 描述 | 长度 |
|------|------|------|
| 数据内容 | 压缩的二进制数据 | 可变 |
| 结束序列 | 传输结束标志 | 3字节 |

### 2.2 结束序列

传输结束标志为固定的3字节序列：`0xFF 0xFF 0xFF`

## 3. 二进制数据格式

硬件端期望接收的是**差分压缩格式的二进制数据**，具体结构如下：

### 3.1 文件头

| 字段 | 描述 | 长度 | 编码方式 |
|------|------|------|----------|
| 总段数 | 轨迹数据的总段数 | 4字节 | 小端序 (Little Endian) |

### 3.2 段结构

每个段包含以下内容：

| 字段 | 描述 | 长度 | 编码方式 |
|------|------|------|----------|
| 段点数 | 当前段的坐标点数量 | 4字节 | 小端序 (Little Endian) |
| 第一个点 | 完整的坐标数据 | 12字节 | 小端序 (Little Endian) |
| 后续点 | 坐标差值数据 | 可变 | 可变长度编码 |

### 3.3 坐标数据结构

#### 第一个点（完整坐标）：

| 字段 | 描述 | 长度 | 编码方式 | 单位 |
|------|------|------|----------|------|
| 经度 | 经度值 | 4字节 | 小端序，int32_t | 度×10^6 |
| 纬度 | 纬度值 | 4字节 | 小端序，int32_t | 度×10^6 |
| 海拔 | 海拔值 | 4字节 | 小端序，int32_t | 米 |

#### 后续点（差值）：

每个后续点由三个差值组成，每个差值使用可变长度编码：

| 字段 | 描述 | 长度 | 编码方式 |
|------|------|------|----------|
| 经度差值 | 与前一点的经度差 | 1-5字节 | 可变长度编码 |
| 纬度差值 | 与前一点的纬度差 | 1-5字节 | 可变长度编码 |
| 海拔差值 | 与前一点的海拔差 | 1-5字节 | 可变长度编码 |

## 4. 可变长度编码规则

差值采用类似于UTF-8的可变长度编码：

1. **字节结构**：
   - 每个字节的最高位表示是否继续（1=继续，0=结束）
   - 第一个字节的第6位表示符号（1=负数，0=正数）
   - 数据位：第一个字节使用低6位，后续字节使用低7位

2. **编码示例**：
   - 小数值：使用较少字节
   - 大数值：使用较多字节

## 5. 数据传输流程

1. 小程序端：
   - 将轨迹数据转换为差分压缩格式
   - 按照上述二进制格式组织数据
   - 在数据末尾添加结束序列 `0xFF 0xFF 0xFF`
   - 通过串口发送给硬件端

2. 硬件端：
   - 接收二进制数据
   - 检测结束序列 `0xFF 0xFF 0xFF`
   - 保存接收到的数据到SD卡
   - （可选）如果启用了解压缩功能，进行解压处理

## 6. 注意事项

1. **数据大小**：确保数据大小不超过硬件端的缓冲区处理能力
2. **结束标志**：必须在数据末尾添加 `0xFF 0xFF 0xFF` 结束标志
3. **编码格式**：严格按照小端序和可变长度编码规则
4. **校验**：当前版本不要求校验和（KML_ENABLE_CRC = 0）

## 7. 示例数据结构

```
[总段数 (4字节)]
[段1点数 (4字节)][第一个点坐标 (12字节)][点2差值 (可变)][点3差值 (可变)]...
[段2点数 (4字节)][第一个点坐标 (12字节)][点2差值 (可变)][点3差值 (可变)]...
...
[结束序列 (3字节: 0xFF 0xFF 0xFF)]
```

## 8. 实现建议

小程序端实现步骤：
1. 收集轨迹坐标数据
2. 将坐标数据转换为差分编码格式
3. 按照上述二进制格式组织数据
4. 添加结束序列
5. 通过串口发送给硬件端

硬件端会自动处理数据接收和存储，无需额外配置。
